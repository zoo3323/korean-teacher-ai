# 빌딩 리포트 — 국어 교사용 AI 지문 분석 웹앱

---

## 1. 설정한 사용자는 누구이며, 해결하려는 구체적인 문제점/불편함은 무엇인가요?

### 타깃 사용자

**중·고등학교 국어 교사** (내신 및 수능 대비 수업 담당)

### 구체적인 문제점

| # | 문제 |
|---|------|
| 1 | 시·소설·수필 등 지문을 교과서나 참고서에서 스캔/촬영한 이미지 상태로 보유하고 있어, 분석 작업에 매번 수작업이 필요하다. |
| 2 | 행별 표현 기법·시어 의미·출제 포인트를 주석으로 달고, 연/단락 구조를 정리하는 참고서 수준의 분석에 교사 1인이 수 시간을 소비한다. |
| 3 | 수능·내신 스타일의 문제를 직접 출제할 때 객관식 선지 구성, 오답 근거 작성 등이 반복적이고 시간 소모가 크다. |
| 4 | 분석 결과물을 학생 배포용 프린트(PDF/Word)로 변환하는 별도 작업이 필요하다. |

---

## 2. 문제를 해결하기 위한 솔루션의 핵심 기능은 무엇인가요?

### 핵심 기능 5가지

**① OCR 텍스트 추출**
- 지문 이미지를 업로드하면 Claude Vision API가 원문 텍스트를 자동 추출
- 추출 후 교사가 직접 편집·교정 가능

**② AI 참고서 스타일 분석**
- 추출된 텍스트를 Claude가 수십 년 경력 국어 교사 페르소나로 분석
- 행별 주석(literary/meaning/tip/teacher), 연·단락 구조, 형식 분석, 표현 기법 목록, 어휘 해설을 JSON으로 구조화
- 결과를 참고서 레이아웃으로 시각화하며, 교사가 직접 편집 가능

**③ 캔버스 직접 분석**
- Fabric.js 기반 캔버스로 지문 이미지 위에 자유롭게 그림·필기·메모 추가
- 수업 판서 스타일 그대로 작업 가능

**④ AI 문제 자동 생성**
- 객관식 5지선다(수능 스타일), 서술형/단답형, 빈칸 추론, 어휘·문법 문제 4가지 유형 지원
- AI 분석 결과를 컨텍스트로 활용해 주제·표현 기법 기반 심화 문제 출제

**⑤ 결과물 내보내기**
- PDF, DOCX 형식으로 분석 결과·문제지 다운로드
- 프로젝트 shareToken 기반으로 브라우저에서 세션 유지 및 공유 가능

## 3. 이 솔루션이 도입되었을 때 기대되는 개선점이 무엇인가요?

| 항목 | 도입 전 | 도입 후 |
|------|---------|---------|
| 지문 분석 시간 | 교사 1인 기준 2~4시간 | AI 분석 1~2분 + 교사 검수 10~20분 |
| 문제 출제 시간 | 문제 유형별 30분~1시간 | 유형·개수 선택 후 AI 생성 1~2분 |
| 분석 품질 일관성 | 교사 역량·컨디션에 따라 편차 발생 | 참고서 수준의 표준화된 분석 구조 |
| 배포 자료 제작 | 별도 문서 편집 툴 필요 | 클릭 한 번으로 PDF/DOCX 출력 |
| 수업 준비 부담 | 반복 작업으로 소진 | 핵심 교육 설계에 집중 가능 |

## 4. 이 프로젝트에서 사용할 AI 도구와 모델은 무엇이며, 선택한 이유는 무엇인가요?

### 사용 도구 및 모델

| 역할 | 도구 / 모델 | 선택 이유 |
|------|------------|-----------|
| **개발 보조** | Claude Code (claude-sonnet-4-6) | 코드베이스 전체 컨텍스트 파악, 파일 읽기·수정·실행을 단일 세션에서 처리. 프로젝트 구조 분석부터 버그 수정까지 일관된 흐름 유지 |
| **OCR / 이미지 분석** | Claude claude-sonnet-4-6 (Vision) | 한국어 지문 이미지에서 행 단위 텍스트 추출 정확도가 높음. 이미지·텍스트 멀티모달 처리를 단일 API로 해결 |
| **지문 분석 / 문제 생성** | Claude claude-sonnet-4-6 | 한국어 문학 도메인 이해도가 높고, 복잡한 JSON 스키마를 지시에 따라 정확히 준수하는 instruction-following 성능이 우수함. 수능·내신 출제 스타일에 대한 사전 학습 수준이 실용적 |

### Claude를 단일 공급자로 선택한 이유

- OCR → 분석 → 문제 생성의 파이프라인을 **동일 API 클라이언트**(`@anthropic-ai/sdk`)로 통일해 복잡도 감소
- Prompt Caching 활용으로 반복 호출 비용 절감 가능
- 시스템 프롬프트 수준의 역할 지정(페르소나)이 한국어 교육 도메인에서 일관된 결과를 보임

## 5. 각 AI 도구별 활용 전략

### 5-1. Claude Code — 개발 보조 에이전트

**페르소나**: 프로젝트 전체를 아는 시니어 풀스택 엔지니어

**핵심 전략**
- 파일 읽기 → 문제 진단 → 수정 → 검증의 사이클을 단일 대화에서 완결
- 서버 실행, 로그 확인, API 테스트(`curl`)를 Bash 도구로 직접 수행해 "코드만 수정하고 끝"이 아닌 실제 동작 검증까지 포함
- 환경 설정(nvm, prisma, .env 등) 이슈는 원인을 먼저 진단 후 최소한의 변경으로 수정

**에이전트 구성**
```
사용자 요청
  └─ Claude Code (메인)
       ├─ Glob / Grep  → 파일 구조 파악
       ├─ Read         → 코드 이해
       ├─ Edit / Write → 코드 수정
       └─ Bash         → 빌드·실행·검증
```

**제약 사항**
- 불필요한 파일 생성 금지 (기존 파일 우선 수정)
- 요청 범위 밖의 리팩토링·기능 추가 금지
- 보안에 민감한 키(.env)는 출력에서 마스킹

---

### 5-2. Claude API (Runtime) — OCR 에이전트

**페르소나**: 한국어 원문 판독 전문가

**핵심 전략**
- `src/app/api/documents/[id]/extract/route.ts`에서 이미지 URL을 Base64로 변환 후 Vision API에 전달
- 프롬프트: "원문 그대로 행 구분을 유지해서 추출, 추측으로 내용을 보완하지 말 것" 지시
- 추출 결과는 DB에 저장하고 교사가 직접 편집 가능하도록 `ExtractedTextEditor` 제공

**데이터 흐름**
```
이미지 업로드(public/uploads)
  → /api/documents/[id]/extract (POST)
  → Claude Vision API
  → extractedText → DB 저장
  → 클라이언트 상태 업데이트
```

---

### 5-3. Claude API (Runtime) — 분석 에이전트

**페르소나**: 수십 년 경력의 국어 교사

**핵심 전략**
- `src/lib/prompts.ts`의 `getAnalysisMessages()`에서 엄격한 JSON 스키마를 프롬프트에 포함
- 행(line) 단위 분석 + callout 타입(literary/meaning/tip/teacher) 4종으로 구조화
- 응답은 JSON만 허용("다른 텍스트는 절대 포함하지 마세요") → 파싱 실패 리스크 최소화
- 분석 결과를 `TextbookAnalysisView`에서 참고서 레이아웃으로 렌더링, 편집 모드 지원

**데이터 흐름**
```
extractedText (DB)
  → /api/documents/[id]/analyze (POST)
  → getAnalysisMessages() → Claude API
  → analysisJson (JSON) → DB 저장
  → TextbookAnalysisView 렌더링
```

---

### 5-4. Claude API (Runtime) — 문제 생성 에이전트

**페르소나**: 수능·내신 전문 출제 위원

**핵심 전략**
- 문제 유형(MULTIPLE_CHOICE / SHORT_ANSWER / FILL_IN_BLANK / VOCABULARY)별 스키마 분기
- `analysisJson`이 있을 경우 분석 컨텍스트를 프롬프트에 주입 → 표현 기법·구조 기반 심화 문제 출제
- 객관식은 반드시 5개 선지 + 정답 번호(1~5) 형식 강제

**데이터 흐름**
```
(extractedText + analysisJson) 선택
  → /api/documents/[id]/questions (POST)
  → getQuestionMessages() → Claude API
  → questionsJson → QuestionSet DB 저장
  → QuestionGenerator 렌더링
```

## 6. 토큰 낭비 최소화 및 유지보수성·재현성 향상 전략

### 토큰 최소화

| 전략 | 적용 방식 |
|------|-----------|
| **JSON only 응답 강제** | 프롬프트 마지막에 "다른 텍스트는 절대 포함하지 마세요" 명시 → 불필요한 설명 토큰 제거 |
| **callout 길이 제한** | 30자 이내, summary 15자 이내 등 필드별 글자 수 제약을 프롬프트에 명시 |
| **분석 컨텍스트 선택적 주입** | 문제 생성 시 `analysisContext`가 있을 때만 프롬프트에 추가 (`getQuestionMessages`의 옵셔널 파라미터) |
| **단계별 API 호출** | OCR → 분석 → 문제 생성을 사용자 트리거로 분리해 불필요한 자동 호출 방지 |

### 유지보수성

| 전략 | 적용 방식 |
|------|-----------|
| **프롬프트 중앙화** | 모든 프롬프트를 `src/lib/prompts.ts` 단일 파일에서 관리 → 수정 시 한 곳만 변경 |
| **JSON 스키마 고정** | `analysisJson`의 타입을 `TextbookAnalysisData`로 TypeScript 타입 정의 → 컴파일 타임 오류 검출 |
| **DB 스키마 단순화** | Project → Document → QuestionSet 3계층 구조로 확장 없이 유지 가능 |
| **환경변수 분리** | `.env`에 API 키·DB URL 집중 관리, 코드에 하드코딩 없음 |

### 재현성

| 전략 | 적용 방식 |
|------|-----------|
| **DB 상태 유지** | 분석 결과·캔버스 상태·문제를 모두 SQLite에 저장 → 페이지 새로고침·재방문 후에도 동일 결과 복원 |
| **shareToken 기반 세션** | localStorage + URL 파라미터로 프로젝트를 식별해 링크 공유만으로 동일 워크스페이스 재현 |
| **편집 가능 분석 결과** | AI 출력이 마음에 들지 않으면 교사가 직접 수정 후 저장 → 재분석 없이 수정된 결과 유지 |
